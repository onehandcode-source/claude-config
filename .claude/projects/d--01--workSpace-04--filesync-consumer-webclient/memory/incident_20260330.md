1. **size=0MB HEAD 응답**: iQueue 메시지의 HEAD 응답 Content-Length=0 → 파일 미기록 or 0바이트 상태 의심
2. **09:27 BEFORE response**: ewas2(`10.176.62.212:8082`)가 응답 시작 전 연결 차단 — ewas2 프로세스 일시 불안정
3. **RECOVERY_AUDIT 에러 (별개)**: `Not found FileQ cNo=160591476/160591477` at 08:37 — DB에 없는 cNo가 recoveryQueue로 유입

## Producer 측 발행 구조 (2026-03-31 확인)

Producer 서버는 `SingoController.singoWriteForIllegalV2`에서:
1. 파일 temp → 실제 경로 `FileUtils.copyFile()` (동기)
2. DB DML: `TBL_SINGO` INSERT → `TBL_FILEQ` INSERT (파일 복사 완료 후)
3. `ImgCompressThread.start()` — **비동기** 소켓으로 외부 이미지 압축 서버에 신호 전달
4. 외부 이미지 압축 서버 → RabbitMQ 메시지 발행

**근본 원인 특정**: 타이밍 문제는 `ImgCompressThread`가 신호를 보낸 시점에 NAS 파일 네트워크 동기화가 완전히 끝나지 않은 경우 발생. 외부 이미지 압축 서버가 `TBL_FILEQ` INSERT 직후 즉시 RabbitMQ 메시지를 발행하면, Consumer가 아직 NAS에 파일이 없는 시점에 다운로드를 시도 → `PrematureCloseException`. 컨트롤러 자체의 파일→DB 순서는 올바름.
